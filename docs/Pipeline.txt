1) Projektweite Invarianten (nicht verhandelbar)
1.1 Alpha‑Konvention: Straight Alpha (global)

Alle Texturen/Frames/Zwischenergebnisse: Straight Alpha
Kein globales Premultiply
Falls einzelne Operationen intern premultiplied rechnen müssen: nur lokal im Shader, danach Rückführung auf Straight.


Hinweis: In HAP‑Workflows wird u. a. „straight alpha“ als Erwartung erwähnt; die Festlegung auf Straight ist damit kompatibel. [renderheads.com]

1.2 Farbraum‑Konvention: Linear (global)

Alle Berechnungen (Effekte, Blendmodes, Composite, Tap‑Combines) erfolgen in Linear.
Ein-/Ausgabe in sRGB ist nur eine I/O‑Frage (Decoding/Encoding), nicht der Arbeitsfarbraum.
Jede Quelle wird vor Effekt‑Stack auf Linear normalisiert.
Tap/Slice Outputs sind standardmäßig Linear + Straight Alpha, außer ein Output ist explizit „Display‑Output“ mit sRGB‑Encoding.

Konsequenz: SampleSource() liefert immer Linear RGBA (Straight).
1.3 Einheitliches Render‑Modell (für alle Quellen)
SourceNode → Normalize/SampleSource → EffectStackPass → CompositePass → Output

SourceNode liefert Texturen + Metadaten (Planes, Typ).
SampleSource() ist die einzige Stelle mit formatabhängiger Logik.
Effekt‑Stack und Compositing sind danach identisch.

1.4 Transform‑Semantik im Stack: Resample

Transform kann überall im Stack stehen.
Bedeutung: UV transformieren und erneut aus der Source sampeln (Resample).
So bleibt freie Reihenfolge in einem Pass definierbar (Transform kostet zusätzliche Samples).

3) Quellen (SourceNodes)
3.1 RAW Source (Startzustand: NumPy/memmap)
Input: RGBA8 Straight Alpha, typischerweise als memmap / contiguous Framefile.
Pflichtfeatures

Contiguous Storage pro Clip (berechenbare Offsets, schnelles Scrubbing)
Prefetch Thread + Ringbuffer pro Player
Asynchroner Uploadpfad (Double/Triple Buffer Pattern)
Texture Reuse (keine Neuallokation pro Frame)

Farbraum

RAW‑Frames müssen deklarieren, ob sie sRGB oder Linear repräsentieren.
Normalisierung auf Linear erfolgt in SampleSource().

3.2 HAP Q Alpha Source (Migration/Ziel)
HAP‑Kontext (für Integration relevant)

HAP ist auf hochperformantes Echtzeit‑Playback ausgerichtet, Tradeoff sind sehr große Dateien (I/O‑Limit). [hap.video], [renderheads.com], [derivative.ca]
HAP Q / HAP Q Alpha verwenden YCoCg und benötigen beim Rendern eine Shader‑Konvertierung; Q Alpha kombiniert zusätzlich die Alpha‑Plane. [hap.video]
Referenzdokumentation beschreibt die Frame‑Struktur und externe Kompressionsschemata (S3TC/BCn etc.). [github.com], [hap.video]

Pflichtfeatures

Demux Container (MOV/AVI) → Frames extrahieren
Decode (MVP: RGBA ok; Ziel: getrennte Planes "cocgsy" + "alpha")
Prefetch/Ringbuffer (I/O‑lastig)
SampleSource() führt aus:

YCoCg→RGB Dekodierung (HAP Q/QAlpha)
Alpha combine
ggf. sRGB→Linear (falls Source als sRGB markiert)
Ergebnis: Linear + Straight Alpha




Hinweis: Support für Q Alpha kann je nach Toolchain variieren; früh mit echten Dateien testen. [renderheads.com], [legacy-hel...sguise.one]

3.3 ISF Source (optional)
ISF ist ein Shader‑Format inkl. Metadaten, gedacht als wiederverwendbare Shader‑Ressource. 
Regel: ISF wird in ein FBO gerendert, Output ist RGBA Straight Alpha; Farbraum deklarieren, dann Normalisierung in SampleSource(). [docs.vidvox.net]

15) Performance‑Policy & Human‑Decision Gate (Dev‑Maschinen / Zielabweichung)
15.1 Grundsatz
Wenn das 30‑fps Ziel nicht erreicht wird (insb. auf Dev‑Maschinen), darf das System nicht automatisch Architektur/Qualität/Funktionsumfang ändern. Stattdessen muss ein Human‑Decision Gate ausgelöst werden: Der Mensch entscheidet, ob der aktuelle gemessene Wert (z. B. 22 fps) in dieser Umgebung akzeptiert wird.
15.2 Trigger (wann gefragt werden muss)
Gate auslösen bei:

Sustained FPS Drop: Average < 30 fps über konfigurierbares Fenster (z. B. 3–10 s) unter Tier‑A‑Konfiguration
wiederholte Budgetverletzung (Frame time > 33.3 ms) über Fenster
Regression Detection (z. B. >20% Drop ggü. Baseline)

15.3 Prompt‑Inhalt
Frage eindeutig:

„30 fps Ziel wird nicht erreicht. Soll der aktuelle Wert (X fps) für diese Maschine/Session verwendet werden?“
optional: Dev/Low‑Spec markieren (ja/nein)
optional: Session vs Profil speichern

15.4 Optionen

A) Akzeptieren: aktueller FPS‑Wert wird genutzt (für diese Session/Profil)
B) Nicht akzeptieren: erst nach Zustimmung Vorschläge anbieten (Profiling/Analyse, optionale Preview‑Mode‑Umschaltung etc.)

Light Mode

Minimal‑Buckets:

Render total (Stack+Composite aggregiert)
JIT total (nur wenn aktiv)
Tap/Slice total (aggregiert)


Flags/Zähler:

Interpreter fallback active? (ja/nein + Anteil)
Cache hit/miss grob
Prefetch starvation events


Overhead Ziel: sehr niedrig (typ. <1–3%)
